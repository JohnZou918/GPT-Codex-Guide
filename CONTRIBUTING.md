# 贡献指南

感谢你帮助 GPT-Codex-Guide。项目面向完全零基础读者，因此容易理解、可以验证、说明原因比堆砌术语更重要。

## 基本要求

- 遵守 [社区行为准则](CODE_OF_CONDUCT.md)。
- 中文优先；英文术语第一次出现时补充中文解释。
- 不假设读者懂编程、命令行、路径、仓库或账号权限。
- 每一个操作步骤都解释“为什么这样做”。
- 不提交密码、令牌、邮箱、真实个人路径等敏感信息。

## 文件命名规范

- 章节使用“两位数字-中文标题.md”，例如 03-Windows安装.md。
- 数字代表阅读顺序，不随意跳号或重复。
- 项目级规范文件使用大写英文名称。
- 文件名不使用空格、表情符号或含义不清的缩写。
- 改名时必须同步修正所有链接。

## 图片命名规范

- 图片存放在 images/。
- 推荐格式：章节号-主题-平台-序号.扩展名。
- 示例：03-git-install-windows-01.png。
- 平台使用 windows、macos 或 common。
- 图片不得包含密码、令牌、邮箱或个人信息。
- 每张图片必须有能说明内容的替代文字。

## Markdown 编写规范

- 一个文件只保留一个一级标题。
- 标题按层级使用，不跳级。
- 代码、命令、路径和文件名应使用行内代码格式。
- 操作步骤使用有序列表，并紧接“为什么”说明。
- 表格用于比较，Mermaid 用于流程，不为装饰增加复杂图表。
- 内部链接使用相对路径。
- 避免大段文字，优先使用短段落、步骤和完成检查。

## 章节固定结构

1. 这是什么？
2. 为什么要安装，或为什么存在？
3. 有什么用途？
4. 怎样一步一步操作？

每一步还应说明在哪里操作、为什么这样做、正确结果是什么、结果不同时先检查什么。

## 建议贡献流程

1. 阅读相关章节和 [项目路线图](ROADMAP.md)，避免重复。
2. 一次只处理一个清晰主题。
3. 预览 Markdown，检查标题、表格、流程图和链接。
4. 提交说明写清改了什么以及为什么改。
5. 发起 Pull Request（拉取请求）时说明适用平台和验证方法。

## 提交前检查

- [ ] 中文优先，新术语已经解释。
- [ ] 每一步已经说明原因。
- [ ] 没有敏感信息。
- [ ] 文件和图片命名符合规范。
- [ ] Markdown 可以正常预览。
- [ ] 相对链接指向存在的文件。
