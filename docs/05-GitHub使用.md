# 第5章 GitHub 是什么

> GitHub 不是安装在电脑里的“另一个 Git”，而是一个在线保存项目、展示版本和开展协作的平台。先理解它的作用，再学习具体操作，会容易得多。

---

## 本章目标

学习完成后，你能够：

- 用一句简单的话解释 GitHub。
- 说清 GitHub 与 Git 的区别。
- 理解项目为什么需要在线仓库。
- 说明 GitHub 在 GPT-Codex-Guide 中的作用。
- 识别新手第一次使用 GitHub 时的常见问题。

- **预计学习时间**：20～30 分钟。
- **适合读者**：没有 Git、GitHub 或编程基础的读者。
- **需要的前置知识**：知道项目是一组为同一目标服务的文件。
- **开始前准备**：本章不要求注册账号，也不要求输入命令。

---

## 一、GitHub 是什么？

### 一句话解释

GitHub 是一个在线保存项目版本、展示内容并支持多人协作的平台。

### 什么是“在线平台”？

在线平台是通过互联网提供服务的网站。你可以使用账号访问它，不必把 GitHub 本身理解成一个普通文件夹。

在 GitHub 中，一个项目通常放在 Repository（仓库）里。Repository 中文叫“仓库”，它包含项目文件以及这些文件的版本记录。

例如，GPT-Codex-Guide 仓库中包含：

- README 项目首页。
- docs 目录中的各章文档。
- images、examples 和 templates 等目录。
- PDF 版本。
- 每次重要修改的提交记录。

### GitHub 是网络硬盘吗？

它有在线保存文件的作用，但不只是网络硬盘。

普通网络硬盘主要关心“现在有哪些文件”。GitHub 还关心：

- 文件以前是什么样。
- 谁在什么时候修改过。
- 为什么进行这次修改。
- 哪些修改已经被项目接受。
- 团队正在讨论哪些问题。

因此，GitHub 更像“带历史记录和协作流程的项目空间”。

### 为什么要先理解仓库？

如果把仓库只看成一个普通文件夹，就容易不明白提交、分支和同步的意义。

“提交”是给一组变化建立一次带说明的版本记录。“分支”是从某个版本延伸出的独立工作线。这些概念会在后续章节逐步学习，本章只需要先知道：GitHub 保存的不只是文件，也保存项目变化的过程。

## 二、GitHub 与 Git 有什么区别？

Git 与 GitHub 经常一起出现，所以新手很容易把它们当成同一个工具。

| 比较内容 | Git | GitHub |
|---|---|---|
| 它是什么 | 记录文件变化的版本管理工具 | 在线保存 Git 仓库并支持协作的平台 |
| 主要位置 | 可以在自己的电脑上工作 | 通过互联网提供服务 |
| 没有网络能否使用 | 可以记录本地版本 | 无法进行在线同步 |
| 主要作用 | 提交、比较和管理版本 | 备份、分享、讨论和协作 |
| 是否必须一起使用 | 不必须 | GitHub 主要围绕 Git 仓库工作 |

“版本管理”就是记录文件怎样一步步变化，方便查看历史或找回较早内容。

### 一个生活比喻

可以把 Git 想成一本放在自己桌上的项目记录册。每次完成一个阶段，你就在记录册中写下“这次改了什么”。

GitHub 像一个在线项目档案室。你把记录册中的新内容同步过去，其他获得权限的人就能查看和协作。

这个比喻并不完全等同于真实技术，但能帮助记忆：

- Git 负责记录。
- GitHub 负责在线保存和协作。

### 为什么本章不讲 Git 命令？

因为先理解“为什么需要记录和同步”，再学习命令，才不会把命令当成没有意义的字符。

下一章会专门解释 Git 是什么。本章只建立 GitHub 的整体认识。

## 三、为什么开发者通常会使用 GitHub？

这里的“开发者”不只指程序员。凡是持续建设文档、网站、程序或其他数字项目的人，都可以参与开发。

GitHub 不是法律规定的必需工具，但它已成为非常常见的项目协作方式。学习它有以下价值。

### 1. 项目不会只留在一台电脑上

如果文件只保存在电脑里，电脑损坏、文件误删或换设备时，项目可能丢失。

把提交推送到 GitHub，可以拥有在线副本。“推送”是把电脑中的新提交发送到远程仓库。

为什么重要？因为本地保存和在线保存是两层保护。不过，GitHub 也不应被当成所有资料的唯一备份，重要内容仍应有合适的备份方案。

### 2. 可以清楚展示项目

GitHub 仓库有固定地址，可以展示 README、目录、文件和提交历史。

为什么重要？别人不需要收到一堆散乱附件，就能看到项目当前状态和说明。

### 3. 可以多人协作

团队成员可以在同一个仓库中工作，并通过 Issue 和 Pull Request 交流。

Issue 中文可理解为“问题或任务记录”。Pull Request 常简称 PR，中文叫“拉取请求”，表示请求维护者检查并合并一组修改。

为什么有用？讨论与修改记录放在项目旁边，比散落在聊天记录中更容易追踪。

### 4. 可以保留清楚历史

当项目出现问题时，可以查看最近哪些内容发生变化。

为什么重要？没有历史记录时，人只能猜测“以前是不是好的”。有提交记录后，可以定位到具体版本。

### 5. 可以在不同设备间继续工作

在 Windows 上完成并推送后，可以在 macOS 上取得最新版本继续工作。

为什么重要？GitHub 充当共同的远程仓库，让不同电脑围绕同一份版本历史工作。

## 四、GitHub 能解决什么问题？

| 常见问题 | GitHub 提供的帮助 | 仍需注意 |
|---|---|---|
| 文件只在一台电脑上 | 保存在线仓库 | 重要资料仍要有备份 |
| 不知道改过什么 | 展示提交历史和差异 | 提交说明要写清楚 |
| 多人互相覆盖文件 | 提供分支和合并流程 | 同步前仍要检查状态 |
| 问题散落在聊天中 | 使用 Issue 记录 | 标题和描述要明确 |
| 不知道哪个版本可用 | 通过主分支和发布记录管理 | 团队要约定发布规则 |
| 换电脑无法继续 | 从远程仓库取得项目 | 开始工作前先同步 |
| 项目难以展示 | README 和仓库页面提供入口 | 不要公开敏感信息 |

GitHub 能改善项目管理，但不能自动解决所有问题。

如果提交说明含糊、权限设置错误或团队不遵守流程，仓库仍然会混乱。工具提供能力，真正稳定还需要清楚规则和良好习惯。

## 五、GitHub 在本项目中起什么作用？

GPT-Codex-Guide 的主要目标，是验证 ChatGPT、Codex、GitHub、Windows 与 macOS 的协同开发流程是否简单、稳定、可重复。

GitHub 在这套流程中承担五项作用。

### 1. 保存在线仓库

每章完成后，本地修改会通过 Git 建立提交，再推送到 GitHub。

为什么这样做？本地文件与在线版本可以互相核对，也可以从其他电脑继续项目。

### 2. 展示项目首页

README 是仓库首页说明文件。读者进入项目后，可以看到项目目的、学习路线和章节入口。

为什么重要？一个没有首页说明的仓库，就像一本没有封面和目录的书。

### 3. 保存每章的提交记录

本项目坚持“一次只开发一章，一次只提交一个 Commit”。

Commit 就是提交。每章拥有单一目的的提交，出现问题时更容易知道哪一次变化需要检查。

### 4. 验证协作流程

本项目使用：

1. 用户确定目标。
2. ChatGPT 参与策划。
3. Codex 修改和检查文件。
4. Git 记录版本。
5. GitHub 保存结果。
6. 用户进行人工验收。

每完成一章，就完整走一遍流程。为什么重复？因为只有多次稳定成功，才能证明流程可重复，而不只是偶然成功一次。

### 5. 提供公开学习入口

读者可以在线查看 Markdown 文档，也可以取得最新版 PDF。

为什么有用？同一项目同时保留便于维护的源文档和便于阅读的成品文件。

下面是本项目从本地修改到在线保存的简化流程：

    A["Codex 修改指定文件"] --> B["检查 Markdown 与 PDF"]
    B --> C["Git 创建一次提交"]
    C --> D["推送到 GitHub"]

## 六、新手第一次使用 GitHub 容易遇到什么问题？

### 1. 把 GitHub 当成 Git

**原因**：两个名称经常一起出现。

**解决方法**：记住“Git 在本地记录，GitHub 在线保存和协作”。

### 2. 不知道仓库是否公开

Public 表示公开，Private 表示私有。公开仓库通常可以被他人看到，私有仓库只对获得权限的人开放。

**解决方法**：创建仓库和上传内容前，检查可见性，不把密码、令牌或公司机密放入仓库。

“令牌”在这里指一种登录凭据，应像密码一样保密。

### 3. 以为保存文件就已经上传

保存只改变电脑中的文件；提交记录本地版本；推送才把提交发送到 GitHub。

**解决方法**：记住三个动作：保存、提交、推送。完成后在 GitHub 页面核对。

### 4. 忘记先取得最新版本

如果另一台电脑或其他成员已经推送新内容，而你仍在旧版本上工作，之后可能出现冲突。

“冲突”表示 Git 无法自动判断两组修改应该怎样组合。

**解决方法**：开始工作前先取得远程最新版本，再修改文件。

### 5. 提交说明看不懂

如果每次只写“修改”“更新”，以后无法知道具体做了什么。

**解决方法**：说明对象和目的，例如“发布第五章 GitHub 入门”。

### 6. 不知道网页上的按钮作用

GitHub 页面包含 Code、Issues、Pull requests、Actions 等入口，新手不需要一次学完。

**解决方法**：先认识仓库首页、文件列表和提交历史。需要某项功能时再学习，不要同时打开所有功能。

### 7. 登录或推送失败

可能原因包括账号未登录、权限不足、网络问题或身份验证过期。

“身份验证”是系统确认你是谁的过程。

**解决方法**：保留完整错误提示，确认账号、仓库地址和权限，不要反复输入或公开密码。

### 8. 害怕操作后无法恢复

新手常担心点错按钮会毁掉项目。

**解决方法**：先在练习仓库操作；重要操作前检查仓库名称和分支；不确定时停止并询问。Git 的版本历史可以帮助恢复，但仍要谨慎处理删除和覆盖。

## 八、本章总结

【本章总结】

GitHub 是在线保存项目版本、展示内容并支持协作的平台。

它与 Git 的区别是：

- Git 主要在本地记录文件变化。
- GitHub 主要在线保存 Git 仓库并支持分享和协作。

GitHub 可以帮助项目保存在线副本、展示内容、保留历史、记录问题、支持多人工作，并让不同电脑继续同一个项目。

在 GPT-Codex-Guide 中，每完成一章，都会检查文件、创建一个提交、推送到 GitHub，再由用户人工验收。这让开发流程可以被追踪、重复和验证。
