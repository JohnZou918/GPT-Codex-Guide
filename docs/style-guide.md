# 图片与写作风格指南

本指南统一 GPT-Codex-Guide 的图片、流程图、代码块和 Markdown 风格。所有新章节必须以 [唯一章节模板](../templates/chapter-template.md) 为基础。

## 图片尺寸

- 推荐截图宽度为 1280～1600 像素，保证文字清晰且文件不过大。
- 单张图片建议小于 500 KB；复杂界面图可适当放宽，但应先压缩。
- 优先使用 PNG 保存界面截图；照片或大面积渐变图使用 JPEG 或 WebP。
- 不强行放大低分辨率图片，因为放大不会增加真实清晰度。
- 同一组步骤截图尽量使用一致的窗口尺寸和缩放比例。

## 截图要求

- 只截取完成说明所需的界面，避免无关桌面区域。
- 截图前隐藏邮箱、账号、文件路径、令牌、通知等个人或敏感信息。
- 用箭头或方框标记操作点时，保持颜色和样式一致，不遮挡原文字。
- 每张截图前后都要有文字说明，不能让图片代替关键步骤。
- 替代文字要描述图片内容和操作目的，例如“VS Code 文件菜单中的打开文件夹选项”。
- Windows 与 macOS 界面明显不同时，应分别提供并明确标注平台。
- 工具界面更新后，应核对旧截图是否仍与步骤一致。

## Mermaid 流程图规范

- 使用 flowchart TD 表示自上而下流程，必要时才使用 LR。
- 节点文字使用简短中文；专业术语可在中文后用括号补充英文。
- 判断节点使用菱形，并为分支标注“是”“否”等明确条件。
- 一个流程图建议不超过 12 个节点；复杂流程拆成多个小图。
- 图前说明它要表达什么，图后解释读者应得到的结论。
- 节点包含标点或括号时使用双引号包裹。
- Mermaid 源码必须在 GitHub 预览中正常渲染。

示例：

~~~mermaid
flowchart TD
    A["开始操作"] --> B{"结果正确吗？"}
    B -- "是" --> C["完成"]
    B -- "否" --> D["查看错误提示"]
    D --> A
~~~

## 图片命名规范

格式：章节号-主题-平台-序号.扩展名。

示例：

- 03-git-download-windows-01.png
- 04-vscode-install-macos-02.png
- 06-commit-common-01.png

规则：

- 章节号使用两位数字。
- 主题使用简短小写英文，用连字符连接。
- 平台使用 windows、macos 或 common。
- 序号使用两位数字并按正文出现顺序排列。
- 图片统一存放在项目根目录 images/。
- 改名后必须同步修改所有引用。

## 代码块规范

- 命令和代码使用带语言标识的围栏代码块。
- PowerShell 使用 powershell，macOS 常用终端命令使用 bash，纯文本输出使用 text。
- 代码块前说明在哪里运行，代码块后说明预期结果。
- 不把提示符、密码、令牌或个人路径混入可复制命令。
- 一次只展示完成当前步骤所需的命令。
- 危险或不可撤销命令必须给出醒目警告和安全替代方案。

示例：

~~~powershell
git --version
~~~

正确结果应显示 Git 的版本号；具体数字可能因安装时间不同而变化。

## Markdown 编写规范

- 每个文件只有一个一级标题，标题层级不跳级。
- 新章节必须复制 chapter-template.md，不自行发明另一套结构。
- 中文优先；英文术语第一次出现时给出中文解释，并加入 [术语表](glossary.md)。
- 段落保持简短，步骤使用有序列表，比较信息使用表格。
- 每一步解释操作、原因、可能问题、解决方法和验证方法。
- 内部链接使用相对路径，并使用有意义的链接文字。
- 不使用“点击这里”作为唯一链接说明。
- 提示、警告和完成标准应清晰区分。
- 不使用大量加粗、表情符号或装饰性内容干扰阅读。
- 提交前预览 Markdown，并检查链接、表格和 Mermaid。
